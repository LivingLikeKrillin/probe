# 프로덕트 변경 안전망 프레임워크 (Product Change Safety Net Framework)

> **버전**: v1.0 draft
> **대상**: 5~15명 규모의 프로덕트 팀 (기획 · 디자인 · 프론트엔드 · 백엔드 · QA)
> **전제**: AI 도구(Claude Code 등) 연동을 기본으로 설계

---

## 0. 이 프레임워크가 해결하는 문제

프로덕트를 운영하다 보면, 기능 하나를 바꾸는 데 코드보다 **커뮤니케이션과 누락**이 더 큰 비용을 만든다.

| 반복되는 문제 | 근본 원인 |
|---------------|-----------|
| "이 상태일 때 화면이 어떻게 되죠?" — 개발 중 발견 | 엣지 케이스를 착수 전에 합의하지 않음 |
| "API 응답이 스펙이랑 다른데요" — 연동 단계에서 발견 | API 계약이 암묵적이거나, 변경이 전파되지 않음 |
| "토큰 바꿨는데 어디가 깨졌는지 모르겠어요" — 배포 후 발견 | UI 변경의 영향 범위를 자동으로 추적하지 않음 |
| "리뷰가 3일째 안 달려요" — PR이 쌓이며 발견 | PR이 너무 크거나, 리뷰에 필요한 증거가 부족 |
| "그건 나중에 해도 되지 않나요?" — 런칭 직전 발견 | 예외를 허용하되 추적하지 않음 |

이 프레임워크는 **사람이 반복적으로 놓치는 구간을 시스템이 먼저 짚어주는 환경**을 만든다.

### 설계 원칙

1. **자동으로 잡을 수 있는 건 사람에게 맡기지 않는다** — 린트, VRT, 스키마 검증, 영향 분석은 CI가 수행한다.
2. **모든 걸 자동화하겠다는 환상은 갖지 않는다** — 사람의 판단이 필요한 구간(디자인 의도, 비즈니스 결정)은 명확히 남긴다.
3. **판단에 필요한 정보를 시스템이 미리 모아준다** — 리뷰어가 "이게 맞나?"를 판단할 때, 증거(diff/리포트/체크리스트)가 이미 준비되어 있다.
4. **예외는 막지 않되, 추적 가능하게 만든다** — 현실적으로 예외는 발생한다. 단, 반드시 기한과 후속 조치가 붙는다.
5. **AI는 보조가 아니라 파이프라인의 구성원이다** — 체크리스트 검증, PR 코멘트 생성, 누락 탐지를 AI가 수행한다.

## 1. 프레임워크 전체 구조

이 프레임워크는 **세 개의 규정**과 이를 관통하는 **하나의 운영 레이어**로 구성된다.

```
┌─────────────────────────────────────────────────────────────┐
│                    운영 레이어 (Operating Layer)               │
│  예외 관리 · AI 자동화 훅 · 리뷰 정책 · CI 게이트 공통 규칙     │
└──────────┬──────────────────┬──────────────────┬────────────┘
           │                  │                  │
     ┌─────▼─────┐    ┌──────▼──────┐    ┌──────▼──────┐
     │  규정 ①   │    │   규정 ②    │    │   규정 ③    │
     │ 상태 케이스│    │  API 계약   │    │  UI 변경    │
     │ 사전 정의  │    │  운영 규정   │    │  운영 규정   │
     │           │    │            │    │            │
     │ 기획·디자인│    │ 프론트·백엔드│    │ 디자인·프론트│
     │ ·개발 합의 │    │  간 계약    │    │  엔드 품질   │
     └─────┬─────┘    └──────┬──────┘    └──────┬──────┘
           │                  │                  │
           ▼                  ▼                  ▼
     착수 전 합의        개발 중 동기화       PR 단위 검증
     (Kick-off)        (Implementation)      (Review/Merge)
```

### 1.1 세 규정의 역할 요약

| 규정 | 핵심 질문 | 시점 | 주체 |
|------|-----------|------|------|
| ① 상태 케이스 사전 정의 | "이 기능에서 사용자가 만날 수 있는 모든 상태를 정의했는가?" | 개발 착수 **전** | 기획 + 디자인 + 개발 |
| ② API 계약 운영 | "프론트와 백엔드가 같은 스펙을 보고 있는가? 변경은 안전하게 전파되는가?" | 개발 **중** | 프론트엔드 + 백엔드 |
| ③ UI 변경 운영 | "토큰/컴포넌트/화면 변경이 안전하고, 영향 범위가 확인되었는가?" | PR/리뷰/**머지** 시 | 디자인 + 프론트엔드 |

### 1.2 규정 간 의존 관계

```
① 상태 케이스 사전 정의
    │
    ├──▶ ② API 계약: 상태 목록이 API 응답 스키마(에러 코드, 빈 목록 등)의 근거
    │
    └──▶ ③ UI 변경: 상태 목록이 Screen Story 필수 케이스(loading/empty/error)의 근거
```

①이 제대로 되지 않으면 ②와 ③이 각자 알아서 엣지 케이스를 정의하게 되고, 서로 안 맞는다.

## 2. 규정 ① — 상태 케이스 사전 정의 규정

### 2.1 목적

개발 착수 전에 **"이 기능에서 사용자가 만날 수 있는 모든 상태"**를 기획·디자인·개발이 함께 정의하고 합의한다. 개발 중에 "이 경우는 어떻게 하죠?"가 발생하는 빈도를 최소화한다.

### 2.2 상태 매트릭스 (State Matrix)

모든 신규 기능/화면/주요 변경은 아래 매트릭스를 **착수 전에** 작성한다.

```md
## State Matrix: [기능/화면 이름]

### 데이터 상태
| 상태 | 조건 | 화면 처리 | API 응답 | 비고 |
|------|------|-----------|----------|------|
| loading | 최초 데이터 요청 중 | 스켈레톤 UI | — | |
| empty | 데이터 0건 | 빈 상태 일러스트 + CTA | `{ items: [], total: 0 }` | |
| success | 데이터 1건 이상 | 목록 렌더링 | `{ items: [...], total: N }` | |
| error (network) | 네트워크 단절 | 재시도 버튼 | timeout / no response | |
| error (server) | 5xx 응답 | 일반 에러 화면 | `{ error: { code, message } }` | |
| error (business) | 비즈니스 규칙 위반 | 인라인 에러 메시지 | `{ error: { code: "LIMIT_EXCEEDED" } }` | |

### 권한/인증 상태
| 상태 | 조건 | 화면 처리 | API 응답 |
|------|------|-----------|----------|
| unauthorized | 로그인 안 됨 | 로그인 유도 | 401 |
| forbidden | 권한 없음 | 권한 안내 | 403 |
| expired | 세션/토큰 만료 | 재인증 유도 | 401 + specific code |

### 입력/인터랙션 상태 (해당 시)
| 상태 | 조건 | 화면 처리 |
|------|------|-----------|
| pristine | 입력 전 | 기본 상태 |
| dirty | 입력 중 | 실시간 검증 표시 |
| validating | 서버 검증 중 | 인라인 로딩 |
| valid / invalid | 검증 완료 | 성공/에러 표시 |
| submitting | 제출 중 | 버튼 비활성 + 로딩 |
| submitted | 제출 완료 | 성공 피드백 |
```

### 2.3 필수 상태 (Mandatory States)

모든 화면/기능은 아래 상태를 **반드시** 정의해야 한다. 하나라도 빠지면 착수 불가.

| 필수 상태 | 이유 |
|-----------|------|
| `loading` | 모든 비동기 데이터 화면에 존재 |
| `empty` | 데이터 0건은 반드시 발생하는 상태 |
| `error` (최소 1종) | 네트워크/서버/비즈니스 중 최소 하나 |
| `success` | 정상 상태 (해피 패스) |

### 2.4 Kick-off 체크리스트

기능/화면 개발 착수 전 Kick-off 미팅(또는 비동기 리뷰)에서 아래를 확인한다.

* [ ] State Matrix가 작성됨
* [ ] 필수 상태 4종(loading/empty/error/success)이 모두 정의됨
* [ ] 각 상태의 **화면 처리**가 명시됨 (디자인 시안 또는 텍스트 설명)
* [ ] 각 상태의 **API 응답**이 명시됨 (응답 구조 또는 HTTP 상태 코드)
* [ ] 기획 · 디자인 · FE · BE 중 최소 3개 역할이 확인(서명/이모지/코멘트)
* [ ] State Matrix 문서 링크가 태스크(Jira/Linear/GitHub Issue)에 첨부됨

### 2.5 변경 시 규칙

개발 중 새로운 상태가 발견되거나 기존 상태가 변경되면:

1. State Matrix를 **먼저 업데이트**한다
2. 변경 사유와 영향 범위를 매트릭스에 기록한다
3. 관련 역할(기획/디자인/FE/BE)에게 **24시간 이내** 공유한다
4. 변경이 API 스펙에 영향을 주면, 규정 ②의 변경 절차를 함께 진행한다
5. 변경이 화면에 영향을 주면, 규정 ③의 Screen Story를 함께 업데이트한다

### 2.6 AI 자동화 포인트

| 자동화 항목 | 도구 | 트리거 |
|-------------|------|--------|
| State Matrix 템플릿 자동 생성 | Claude Code | 이슈/태스크 생성 시 |
| 필수 상태 누락 탐지 | Claude Code hook | PR 생성 시, Screen Story에 필수 상태가 빠져있으면 경고 |
| API 스펙 ↔ State Matrix 불일치 탐지 | CI script | API 스펙의 에러 코드와 State Matrix의 error 상태 비교 |

### 2.7 산출물 위치 (운영 기본값)

* State Matrix 문서: 프로젝트 Wiki 또는 `docs/state-matrices/[feature-name].md`
* 완료된 매트릭스는 레포에 커밋하여 변경 이력을 추적한다

## 3. 규정 ② — API 계약 운영 규정

### 3.1 목적

프론트엔드와 백엔드가 **항상 같은 스펙을 보고 개발**하고, API 변경이 발생하면 **상대방이 반드시 인지**하도록 보장한다.

### 3.2 API 스펙의 Source of Truth

| 항목 | 원본 | 형식 |
|------|------|------|
| API 스펙 정의 | OpenAPI 3.x (또는 GraphQL SDL) | `api/openapi.yaml` 또는 `api/schema.graphql` |
| 스펙 문서 렌더링 | 자동 생성 (Swagger UI / GraphQL Playground) | CI에서 빌드 |
| Mock 서버 | 스펙 기반 자동 생성 | `pnpm api:mock` |

**원칙**: API 스펙 파일이 유일한 원본이다. 구두 합의, Slack 메시지, Notion 메모는 원본이 아니다.

### 3.3 API 스펙 필수 항목

모든 엔드포인트는 아래를 반드시 포함한다.

```yaml
# 최소 필수 항목 (OpenAPI 예시)
paths:
  /resources:
    get:
      summary: 리소스 목록 조회      # 1. 요약
      description: ...               # 2. 상세 설명
      parameters: ...                # 3. 요청 파라미터 (타입/필수/기본값)
      responses:
        200:                          # 4. 성공 응답 스키마
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/ResourceList'
        400:                          # 5. 클라이언트 에러 (최소 1종)
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/ErrorResponse'
        500:                          # 6. 서버 에러
          ...
```

**필수 응답 정의**:

| 항목 | 필수 여부 | 설명 |
|------|-----------|------|
| 성공 응답 (2xx) | 필수 | 응답 스키마, nullable 필드 명시 |
| 빈 목록 응답 | 필수 (목록형) | `items: []`의 구조 명시 |
| 에러 응답 (4xx) | 필수 (최소 1종) | 에러 코드 enum + 메시지 구조 |
| 에러 응답 (5xx) | 필수 | 서버 에러 공통 구조 |
| 페이지네이션 | 필수 (목록형) | cursor/offset 방식 + 파라미터 명시 |

### 3.4 nullable / optional 명시 규칙

API 계약에서 가장 빈번하게 문제가 되는 부분.

```yaml
# 모든 필드는 아래 3가지 중 하나를 명시해야 한다
properties:
  name:
    type: string              # required + non-nullable (기본)
  nickname:
    type: string
    nullable: true            # required + nullable (값이 null일 수 있음)
  bio:                        # optional (필드 자체가 없을 수 있음)
    type: string

# "optional이면서 nullable"은 금지한다 — 혼란의 원인
# 필드가 없을 수 있으면 optional, 있되 비어있을 수 있으면 nullable
```

**린트 규칙**: `pnpm lint:api`에서 nullable과 optional이 동시에 적용된 필드를 감지하면 CI 실패.

### 3.5 API 변경 분류 & 호환성 규칙

| 변경 유형 | 예시 | 호환성 | 필수 조치 |
|-----------|------|--------|-----------|
| **추가 (Additive)** | 새 엔드포인트, 응답에 필드 추가, 새 optional 파라미터 | 호환 | 스펙 업데이트 → 프론트 공유 |
| **수정 (Modification)** | 필드 타입 변경, 필수/선택 변경, 응답 구조 변경 | **비호환** | 버전 관리 또는 점진적 마이그레이션 |
| **삭제 (Removal)** | 엔드포인트 제거, 필드 제거 | **비호환** | Deprecation → 삭제 (10절과 동일 패턴) |
| **동작 변경 (Behavioral)** | 같은 입력에 다른 결과, 정렬 순서 변경, 기본값 변경 | **잠재적 비호환** | 변경 사유 + 영향 분석 필수 |

### 3.6 API 변경 PR 규칙

**라벨**:

| 라벨 | 의미 |
|------|------|
| `api:additive` | 호환 변경 (추가) |
| `api:breaking` | 비호환 변경 (수정/삭제/동작 변경) |
| `api:deprecation` | 폐기 예고 |

**PR 필수 포함 항목**:

```md
## API Change
### Changed endpoints
- `GET /resources` — 응답에 `metadata` 필드 추가

### Compatibility
- breaking: no
- migration guide: (breaking이면 필수)

### Affected clients
- (프론트엔드 영향 범위 — 수동 기재 또는 자동 탐지)

### State Matrix sync
- (관련 State Matrix 업데이트 여부)
```

### 3.7 API Deprecation 생명주기

UI 토큰 Deprecation(규정 ③ 10절)과 동일한 패턴.

```
[Active] → [Deprecated] → [Removed]
```

| 단계 | 스펙 표시 | 런타임 동작 | 최소 유지 기간 |
|------|-----------|-------------|----------------|
| Active | — | 정상 | — |
| Deprecated | `deprecated: true` + `x-replacement` | 정상 동작 + 응답 헤더에 `Sunset` 추가 | **2 스프린트** |
| Removed | 스펙에서 제거 | 410 Gone 또는 404 | — |

### 3.8 스펙 기반 자동화

| 스크립트 | 역할 |
|----------|------|
| `pnpm api:lint` | OpenAPI 스펙 검증 (필수 항목 누락, nullable/optional 규칙, 네이밍 일관성) |
| `pnpm api:mock` | 스펙 기반 Mock 서버 자동 생성 (프론트 개발용) |
| `pnpm api:diff` | Git diff 기반 API 스펙 변경 탐지 → breaking 여부 자동 판별 |
| `pnpm api:codegen` | 스펙 → 프론트엔드 타입/클라이언트 자동 생성 |
| `pnpm api:compatibility` | 이전 버전 스펙과 현재 스펙의 호환성 자동 검사 |

### 3.9 프론트-백엔드 동기화 워크플로

```
BE가 스펙 변경
    │
    ├─ api:diff가 변경 탐지
    │
    ├─ breaking=no → api:codegen으로 타입 자동 갱신 → FE에게 자동 알림
    │
    └─ breaking=yes → 마이그레이션 가이드 필수
                      → FE 확인 전 머지 불가 (리뷰어 매트릭스 적용)
```

### 3.10 AI 자동화 포인트

| 자동화 항목 | 도구 | 트리거 |
|-------------|------|--------|
| API 스펙 변경 요약 생성 | Claude Code hook | API 스펙 파일 변경 PR 생성 시 |
| breaking 변경 영향 분석 | Claude Code + api:diff | PR에 `api:breaking` 라벨 시 |
| State Matrix ↔ API 에러 코드 불일치 탐지 | CI script | API 스펙 또는 State Matrix 변경 시 |
| Mock 서버 자동 갱신 | CI | API 스펙 변경 머지 시 |

## 5. 운영 레이어 — 공통 규칙

세 규정을 관통하는 공통 운영 규칙.

### 5.1 예외 처리 (공통)

모든 규정에서 예외는 동일한 절차를 따른다.

* [ ] 예외 사유를 2줄 이내로 작성
* [ ] 후속 조치 이슈 링크 포함
* [ ] **2 스프린트 이내** 정상화
* [ ] 초과 시 자동 escalation

### 5.2 리뷰 정책 (공통)

| PR 유형 | 최소 리뷰어 | 비고 |
|---------|-------------|------|
| 규정 ① 관련 (State Matrix 변경) | 기획 1 + 개발 1 | 착수 전 합의이므로 머지 게이트 아님 |
| 규정 ② 관련 (API 스펙 변경) | FE 1 + BE 1 | breaking이면 양측 리드 포함 |
| 규정 ③ 관련 (UI 변경) | 규정 ③ 8.1 매트릭스 참조 | scope/breaking에 따라 차등 |

### 5.3 AI 자동화 레이어

이 프레임워크에서 AI(Claude Code 등)가 담당하는 역할.

#### 5.3.1 역할 정의

AI는 **"확인하고 알려주는 역할"**이다. 최종 판단은 사람이 한다.

```
AI가 하는 일                          사람이 하는 일
─────────────                         ─────────────
체크리스트 자동 검증                    예외 허용 여부 판단
누락 항목 탐지 및 경고                  디자인 의도 확인
PR 코멘트 자동 생성                     비즈니스 결정
영향 범위 리포트 생성                   최종 승인
템플릿/보일러플레이트 생성               우선순위 판단
```

#### 5.3.2 Claude Code Hook 설계

```
┌─────────────────────────────────────────────────────┐
│                 Claude Code Hooks                    │
│                                                     │
│  ┌─────────────┐  ┌──────────────┐  ┌────────────┐ │
│  │ pre-commit  │  │ pr-create    │  │ pr-merge   │ │
│  │             │  │              │  │            │ │
│  │ · 직접값    │  │ · DoD 체크   │  │ · 최종 DoD │ │
│  │   사용 탐지  │  │ · 라벨 제안  │  │   재검증    │ │
│  │ · 스펙 린트  │  │ · 누락 경고  │  │ · 예외 추적│ │
│  │             │  │ · 템플릿 채움 │  │   갱신     │ │
│  └─────────────┘  └──────────────┘  └────────────┘ │
└─────────────────────────────────────────────────────┘
```

**Hook별 동작 상세**:

| Hook | 트리거 | 동작 | 실패 시 |
|------|--------|------|---------|
| `pre-commit` | `git commit` | 직접 값 사용 탐지, API 스펙 린트 | 커밋 차단 + 수정 제안 |
| `pr-create` | PR 생성 | DoD 체크리스트 자동 검증, 라벨 제안, 누락 항목 경고, Impact/API Change 섹션 초안 생성 | 경고 코멘트 (차단하지 않음) |
| `pr-merge` | 머지 시도 | 최종 DoD 재검증, 미승인 VRT 확인, 예외 추적 이슈 자동 생성 | 머지 차단 |

#### 5.3.3 AI 자동화 우선순위 (도입 순서)

모든 자동화를 한 번에 도입하지 않는다. 효과 대비 구현 난이도 기준으로 단계적 도입.

| 우선순위 | 자동화 항목 | 효과 | 난이도 |
|----------|-------------|------|--------|
| P0 (즉시) | PR 템플릿 자동 채움 + 라벨 제안 | 높음 | 낮음 |
| P0 (즉시) | DoD 체크리스트 자동 검증 | 높음 | 낮음 |
| P1 (1개월) | API 스펙 변경 탐지 + breaking 판별 | 높음 | 중간 |
| P1 (1개월) | UI 영향 범위 리포트 (ui:impact) | 높음 | 중간 |
| P2 (2개월) | State Matrix 템플릿 자동 생성 | 중간 | 낮음 |
| P2 (2개월) | State Matrix ↔ API 에러 코드 불일치 탐지 | 중간 | 중간 |
| P3 (3개월) | VRT 결과 요약 + 이상 탐지 | 중간 | 높음 |

### 5.4 CI 게이트 통합 파이프라인

세 규정의 CI 체크를 하나의 파이프라인으로 통합한다.

```
PR 생성
  │
  ├─ 1. Install (pnpm install --frozen-lockfile)
  │
  ├─ 2. Static Analysis (병렬)
  │     ├─ pnpm typecheck
  │     ├─ pnpm lint
  │     ├─ pnpm lint:tokens    (토큰 변경 시)
  │     ├─ pnpm lint:api       (API 스펙 변경 시)
  │     └─ pnpm lint:a11y
  │
  ├─ 3. Build
  │     ├─ pnpm storybook:build
  │     └─ pnpm api:mock (API 변경 시)
  │
  ├─ 4. Visual & Contract Test (병렬)
  │     ├─ pnpm vrt:run
  │     └─ pnpm api:compatibility (API 변경 시)
  │
  ├─ 5. Impact Report
  │     ├─ pnpm ui:impact      (UI 변경 시)
  │     └─ pnpm api:diff       (API 변경 시)
  │
  └─ 6. AI Hook: DoD 검증 + PR 코멘트 생성
```

### 5.5 커뮤니케이션 채널 규칙

시스템이 아무리 좋아도 알림이 무시되면 소용없다.

| 이벤트 | 채널 | 대상 | 형식 |
|--------|------|------|------|
| API breaking 변경 PR | Slack #api-changes | FE 전원 | 자동 알림 (bot) |
| VRT diff 20건 초과 | Slack #design-system | Designer + FE Lead | 자동 알림 (bot) |
| State Matrix 변경 | 태스크 코멘트 | 관련 역할 전원 | 수동 (작성자 책임) |
| 예외 PR 머지 | Slack #exceptions | 팀 전원 | 자동 알림 (bot) |
| Deprecation 기한 도래 | Slack #tech-debt | 담당자 | 자동 알림 (weekly) |

## 6. 도입 로드맵

한 번에 세 규정을 모두 도입하지 않는다. 팀에 가장 큰 고통을 주는 구간부터 순서대로.

### Phase 1: 기반 (1~2주)

> 목표: 최소한의 자동 검증이 돌아가는 상태

* [ ] PR 템플릿 통합 (UI + API 공통)
* [ ] CI 파이프라인에 `typecheck` + `lint` + `storybook:build` 고정
* [ ] Claude Code hook: `pr-create` 시 DoD 체크리스트 자동 검증

### Phase 2: UI 규정 (2~4주)

> 목표: 규정 ③이 완전히 동작하는 상태

* [ ] 토큰 산출물 고정 + `design:sync` 구현
* [ ] Foundations 자동 렌더 + VRT(Chromatic) 도입
* [ ] `lint:tokens` + `lint:a11y` + `ui:impact` 구현
* [ ] 리뷰어 승인 매트릭스 적용

### Phase 3: API 규정 (4~6주)

> 목표: 규정 ②가 완전히 동작하는 상태

* [ ] OpenAPI 스펙 파일을 레포에 커밋하는 워크플로 확립
* [ ] `api:lint` + `api:diff` + `api:compatibility` 구현
* [ ] Mock 서버 자동 생성 (`api:mock`)
* [ ] 타입/클라이언트 자동 생성 (`api:codegen`)
* [ ] API 변경 Slack 자동 알림

### Phase 4: 상태 케이스 규정 (6~8주)

> 목표: 규정 ①이 팀 습관으로 정착하는 상태

* [ ] State Matrix 템플릿 도입
* [ ] Kick-off 체크리스트 운영 시작
* [ ] State Matrix ↔ Screen Story 연결 검증
* [ ] State Matrix ↔ API 에러 코드 불일치 자동 탐지

### Phase 5: 통합 & 최적화 (8~12주)

> 목표: 세 규정이 유기적으로 연결되어 동작

* [ ] 세 규정 간 교차 검증 자동화
* [ ] 예외 추적 대시보드
* [ ] 도입 후 회고 → 규정 경량화/강화 판단

## 7. 성공 지표

프레임워크가 잘 동작하는지 판단하는 기준.

| 지표 | 측정 방법 | 목표 |
|------|-----------|------|
| **개발 중 엣지 케이스 발견율** | "이 상태는 어떻게 하죠?" 류 질문 건수 / 스프린트 | 도입 전 대비 **50% 감소** |
| **API 불일치 발생 건수** | FE-BE 간 "스펙이 다르다" 이슈 수 / 스프린트 | 도입 전 대비 **70% 감소** |
| **VRT에서 잡은 의도치 않은 변경** | VRT diff 중 "이건 의도 아님" 비율 | **월 1건 이하** |
| **PR 리뷰 평균 소요 시간** | PR 생성 → 첫 리뷰 코멘트 시간 | **24시간 이내** |
| **예외 PR 비율** | `ui:exception` / 전체 UI PR | **10% 이하** |
| **예외 정상화 기한 준수율** | 기한 내 정상화 PR 머지 비율 | **90% 이상** |

## 8. 규정 자체의 유지보수

규정은 살아있는 문서다. 팀과 함께 진화한다.

### 8.1 개정 주기

* **정기 리뷰**: 분기 1회, 스프린트 회고와 함께
* **즉시 개정**: 규정이 현실과 맞지 않아 예외가 3회 연속 발생한 경우

### 8.2 개정 절차

1. 변경 제안 (누구나 가능)
2. 팀 리뷰 (비동기, 3영업일 이내)
3. 합의 후 문서 업데이트
4. 변경 이력에 기록

### 8.3 경량화 원칙

규정이 팀을 느리게 만들고 있다면 잘못된 것이다.

* 1 스프린트 동안 한 번도 트리거되지 않은 체크는 제거 후보
* CI에서 false positive가 반복되는 린트 규칙은 조정
* "이 규칙 왜 있는지 모르겠다"는 피드백이 2명 이상이면 즉시 재검토
