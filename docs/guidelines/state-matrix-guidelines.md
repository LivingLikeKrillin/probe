# 상태 케이스 사전 정의 규정 (State Matrix Guidelines)

> **프레임워크 위치**: 규정 ① — 착수 전 합의
> **버전**: v1.0 draft
> **대상**: 기획 · 디자인 · 프론트엔드 · 백엔드 (전체 프로덕트 팀)

---

## 0. 목적

개발 착수 전에 **"이 기능에서 사용자가 만날 수 있는 모든 상태"**를 정의하고, 기획·디자인·개발이 같은 그림을 보고 시작한다.

이 규정이 없을 때 반복되는 상황:

| 시점 | 발생하는 문제 | 비용 |
|------|---------------|------|
| 개발 중반 | "로딩 중일 때 화면 어떻게 하죠?" | 디자인 재작업 + 개발 대기 |
| FE-BE 연동 | "이 에러코드 처리가 빠져있네요" | 양쪽 코드 수정 |
| QA 단계 | "데이터 0건일 때 빈 화면이에요" | 핫픽스 |
| 운영 중 | "이 권한 상태를 고려 안 했네요" | 장애 대응 |

**목표**: 위 문제의 발생 빈도를 도입 전 대비 50% 감소시킨다.

---

## 1. 용어 정의

* **State Matrix(상태 매트릭스)**: 하나의 기능/화면이 가질 수 있는 모든 상태를 표로 정리한 문서. 각 상태에 대해 조건, 화면 처리, API 응답을 명시한다.
* **Mandatory State(필수 상태)**: 어떤 기능/화면이든 반드시 정의해야 하는 상태. 누락 시 착수 불가.
* **Kick-off(킥오프)**: 개발 착수 전, State Matrix를 기반으로 기획·디자인·개발이 합의하는 시점. 미팅 또는 비동기 리뷰 모두 가능.
* **State Discovery(상태 발견)**: 개발 중 State Matrix에 없던 상태가 발견되는 것. 규정에 따라 매트릭스를 업데이트하고 전파한다.

## 2. 적용 범위

### 2.1 반드시 State Matrix가 필요한 경우

* 신규 화면/페이지 추가
* 기존 화면에 데이터 소스가 추가되는 기능 변경 (새 API 연동 등)
* 사용자 인터랙션 흐름이 변경되는 경우 (폼 추가, 다단계 프로세스 등)
* 권한/역할 기반 분기가 추가되는 경우

### 2.2 State Matrix가 불필요한 경우

* 단순 텍스트/카피 변경
* 스타일만 변경 (토큰/컴포넌트 외형 수정, 규정 ③으로 커버)
* 버그 수정 (기존 State Matrix에 이미 정의된 상태의 구현 오류)
* 리팩터링 (외부 동작 변화 없음)

### 2.3 판단이 애매한 경우

"이것도 State Matrix를 만들어야 하나?"라는 질문이 생기면, **한 가지만 확인**한다:

> "이 변경으로 인해 사용자가 만날 수 있는 **새로운 상태**가 생기는가?"

Yes → State Matrix 작성. No → 불필요.

## 3. State Matrix 작성법

### 3.1 매트릭스 구조

State Matrix는 아래 **네 개의 레이어**로 구성된다. 모든 레이어가 항상 필요한 것은 아니다.

```
┌──────────────────────────────────────────────┐
│ Layer 1. 데이터 상태 (항상 필수)               │
│   loading / empty / success / error           │
├──────────────────────────────────────────────┤
│ Layer 2. 권한·인증 상태 (해당 시)              │
│   unauthorized / forbidden / expired          │
├──────────────────────────────────────────────┤
│ Layer 3. 입력·인터랙션 상태 (해당 시)          │
│   pristine / dirty / validating / submitting  │
├──────────────────────────────────────────────┤
│ Layer 4. 비즈니스 특수 상태 (해당 시)          │
│   도메인별 고유 상태                           │
└──────────────────────────────────────────────┘
```

### 3.2 Layer 1 — 데이터 상태 (항상 필수)

비동기 데이터를 다루는 모든 화면에 적용. **4개 상태가 필수**이며 하나라도 빠지면 킥오프 통과 불가.

| 상태 | 조건 | 화면 처리 | API 응답 | 필수 |
|------|------|-----------|----------|------|
| `loading` | 데이터 요청 중 | (구체적으로 명시) | — | **필수** |
| `empty` | 데이터 0건 | (구체적으로 명시) | (응답 구조 명시) | **필수** |
| `success` | 데이터 정상 수신 | (구체적으로 명시) | (응답 구조 명시) | **필수** |
| `error` | 요청 실패 | (구체적으로 명시) | (에러 구조 명시) | **필수** |

**"화면 처리" 칸에 들어가야 하는 것**:

구체적인 UI 행동을 적는다. "적절히 처리" 같은 모호한 표현은 금지.

```
# 나쁜 예
| loading | 데이터 요청 중 | 로딩 표시 | — |

# 좋은 예
| loading | 데이터 요청 중 | 스켈레톤 UI (카드 3개 플레이스홀더) | — |

# 좋은 예
| loading | 데이터 요청 중 | 기존 데이터 유지 + 상단 인라인 스피너 | — |
```

**error 상태 세분화**:

error는 최소 1종 필수이나, 실무에서는 아래처럼 세분화하는 것을 권장한다.

| 에러 유형 | 조건 | 화면 처리 | API 응답 |
|-----------|------|-----------|----------|
| `error:network` | 네트워크 단절/타임아웃 | 전체 에러 화면 + 재시도 버튼 | timeout / no response |
| `error:server` | 5xx 응답 | 전체 에러 화면 + 재시도 버튼 | `500` + `{ error: { code, message } }` |
| `error:business` | 비즈니스 규칙 위반 | 인라인 에러 메시지 (화면 유지) | `4xx` + `{ error: { code: "LIMIT_EXCEEDED" } }` |
| `error:not-found` | 리소스 없음 | 404 화면 또는 빈 상태로 폴백 | `404` |

### 3.3 Layer 2 — 권한·인증 상태 (해당 시)

로그인/권한이 관여하는 화면에 적용.

| 상태 | 조건 | 화면 처리 | API 응답 |
|------|------|-----------|----------|
| `unauthorized` | 로그인 안 됨 | 로그인 페이지 리다이렉트 또는 로그인 유도 모달 | `401` |
| `forbidden` | 로그인됨 + 권한 없음 | 권한 안내 화면 (어떤 권한이 필요한지 명시) | `403` |
| `expired` | 세션/토큰 만료 | 재인증 유도 (자동 갱신 시도 후 실패 시) | `401` + specific code |

**판단 기준**: 이 화면에 "로그인 안 한 사용자"나 "권한 없는 사용자"가 접근할 수 있는가? 가능하면 Layer 2 필수.

### 3.4 Layer 3 — 입력·인터랙션 상태 (해당 시)

폼, 검색, 필터 등 사용자 입력이 핵심인 화면에 적용.

| 상태 | 조건 | 화면 처리 |
|------|------|-----------|
| `pristine` | 아무 입력 없음 | 기본 상태, 검증 메시지 없음 |
| `dirty` | 입력 시작됨 | 실시간 검증 표시 (해당 시) |
| `validating` | 서버 사이드 검증 중 | 인라인 로딩 인디케이터 |
| `valid` | 검증 통과 | 성공 표시 (체크마크 등) |
| `invalid` | 검증 실패 | 에러 메시지 + 필드 하이라이트 |
| `submitting` | 제출 요청 중 | 제출 버튼 비활성 + 로딩 |
| `submitted:success` | 제출 성공 | 성공 피드백 (토스트/리다이렉트/화면 갱신) |
| `submitted:error` | 제출 실패 | 에러 메시지 + 재시도 가능 상태로 복귀 |

**주의**: `submitting` 상태에서 **중복 제출 방지** 방법을 반드시 명시한다 (버튼 비활성화? 요청 디바운스? 서버 멱등성?).

### 3.5 Layer 4 — 비즈니스 특수 상태 (해당 시)

도메인에 따라 고유한 상태가 존재한다. 패턴화할 수 없으므로 예시로 안내.

```md
# 예시: 결제 화면
| 상태 | 조건 | 화면 처리 | API 응답 |
|------|------|-----------|----------|
| `payment:pending` | 결제 승인 대기 | 대기 화면 + 폴링 | `{ status: "pending" }` |
| `payment:processing` | PG사 처리 중 | 프로그레스 화면 (이탈 경고) | `{ status: "processing" }` |
| `payment:success` | 결제 완료 | 완료 화면 + 주문 정보 | `{ status: "success", orderId }` |
| `payment:failed` | 결제 실패 | 실패 사유 + 재시도/다른 수단 | `{ status: "failed", reason }` |
| `payment:cancelled` | 사용자 취소 | 이전 화면으로 복귀 | `{ status: "cancelled" }` |

# 예시: 실시간 협업 문서
| 상태 | 조건 | 화면 처리 |
|------|------|-----------|
| `collab:connected` | WebSocket 연결 중 | 실시간 커서/편집 표시 |
| `collab:disconnected` | 연결 끊김 | "연결 끊김" 배너 + 로컬 편집 유지 |
| `collab:reconnecting` | 재연결 시도 중 | "재연결 중..." 표시 |
| `collab:conflict` | 충돌 발생 | 충돌 해결 UI |
```

### 3.6 상태 간 전이 (State Transitions)

복잡한 흐름이 있는 기능에서는 상태 간 전이를 명시한다. **모든 기능에 필수는 아니며**, 아래 조건에 해당하면 작성한다.

**전이 다이어그램이 필요한 경우**:
* 상태가 5개 이상
* 비선형 흐름 (되돌아가기, 분기, 병렬 상태)
* 타이머/폴링 기반 전이

```
# 텍스트 기반 전이 표기 (Mermaid 또는 텍스트)
loading → success | error
success → loading (새로고침 시)
error → loading (재시도 시)
empty → loading (필터 변경 시)
```

필수는 아니지만, 작성하면 QA 테스트 케이스 도출이 훨씬 쉬워진다.

## 4. 필수 상태 (Mandatory States)

### 4.1 절대 필수 (모든 비동기 화면)

| 상태 | 이유 | 빠지면 생기는 문제 |
|------|------|---------------------|
| `loading` | 네트워크는 항상 지연을 가진다 | 흰 화면 깜빡임, 사용자 혼란 |
| `empty` | 데이터 0건은 반드시 발생한다 | 빈 화면 방치, 사용자 이탈 |
| `success` | 정상 흐름 (해피 패스) | — (이건 안 빠지지만 명시적으로 정의) |
| `error` (최소 1종) | 네트워크/서버는 반드시 실패한다 | 무한 로딩, 사용자 막힘 |

### 4.2 조건부 필수

| 조건 | 추가 필수 상태 |
|------|----------------|
| 로그인/권한이 관여 | `unauthorized`, `forbidden` |
| 폼/입력이 핵심 | `pristine`, `invalid`, `submitting` |
| 페이지네이션/무한 스크롤 | `loading:more` (추가 로딩), `end-of-list` |
| 실시간 데이터 (WebSocket 등) | `connected`, `disconnected`, `reconnecting` |
| 낙관적 업데이트 (Optimistic UI) | `optimistic`, `rollback` |

## 5. Kick-off (착수 합의) 절차

### 5.1 Kick-off 형식

**동기(미팅)** 또는 **비동기(문서 리뷰)** 모두 가능. 팀 상황에 맞게 선택.

| 형식 | 언제 적합한가 | 소요 시간 |
|------|---------------|-----------|
| 동기 미팅 | 새로운 도메인, 복잡한 흐름, 의견 분분할 때 | 30~60분 |
| 비동기 리뷰 | 익숙한 패턴, 변경 범위가 명확할 때 | 1~2영업일 |

### 5.2 Kick-off 전 준비 (작성자 책임)

작성자는 보통 기능을 담당하는 **FE 또는 기획자**이다. 누가 작성해도 무방하나, 한 명이 초안을 잡고 나머지가 리뷰한다.

* [ ] State Matrix 초안 작성 (3절 참조)
* [ ] 필수 상태 4종 포함 확인
* [ ] Layer 2~4 해당 여부 판단 및 작성
* [ ] 디자인 시안과 매트릭스의 상태가 1:1 대응되는지 확인
* [ ] 태스크(Jira/Linear/GitHub Issue)에 State Matrix 링크 첨부

### 5.3 Kick-off 체크리스트 (합의 확인)

아래 모든 항목이 충족되어야 개발에 착수한다.

* [ ] State Matrix가 작성됨
* [ ] 필수 상태 4종(loading/empty/error/success)이 모두 정의됨
* [ ] 각 상태의 **화면 처리**가 구체적으로 명시됨 ("적절히 처리" 금지)
* [ ] 각 상태의 **API 응답**이 명시됨 (응답 구조 또는 HTTP 상태 코드)
* [ ] "화면 처리"와 디자인 시안이 일치하거나, 시안 제작 계획이 있음
* [ ] 기획 · 디자인 · FE · BE 중 **최소 3개 역할**이 확인
* [ ] 확인 방법: 문서에 코멘트/이모지/서명 (형식 자유, 기록이 남으면 됨)

### 5.4 확인의 의미

각 역할이 확인할 때 보는 관점:

| 역할 | 확인하는 것 |
|------|-------------|
| **기획** | 비즈니스 시나리오가 빠짐없이 반영되었는가, 엣지 케이스가 사용자 경험상 합리적인가 |
| **디자인** | 모든 상태에 대한 화면 처리가 디자인 가능한가, 시안 제작 범위가 명확한가 |
| **FE** | 화면 처리가 기술적으로 구현 가능한가, 상태 전이가 합리적인가 |
| **BE** | API 응답 구조가 구현 가능한가, 에러 코드가 충분한가, 성능 이슈가 없는가 |

### 5.5 Kick-off 없이 착수 가능한 예외

* 긴급 핫픽스 (운영 장애 대응): 착수 후 **1영업일 이내** State Matrix를 소급 작성
* 프로토타입/PoC: State Matrix 대신 **"확인된 미정의 상태 목록"**을 태스크에 기록

## 6. 개발 중 상태 발견 (State Discovery)

### 6.1 문제 상황

Kick-off에서 아무리 꼼꼼히 정의해도 개발 중에 새로운 상태가 발견된다. 이건 정상이다. 문제는 발견된 상태를 **방치하거나, 구두로만 합의하는 것**이다.

### 6.2 State Discovery 절차

```
개발자가 새 상태 발견
    │
    ├─ 1. State Matrix에 상태 추가 (상태 이름 + 조건만이라도)
    │
    ├─ 2. 영향 판단
    │     ├─ 화면 처리만 추가 → 디자인에 공유
    │     ├─ API 응답 변경 필요 → BE에 공유 + 규정 ② 연동
    │     └─ 양쪽 모두 → 디자인 + BE 공유
    │
    ├─ 3. 24시간 이내 관련 역할에 공유
    │     └─ 채널: 태스크 코멘트 (기록이 남는 곳)
    │
    └─ 4. 합의 후 State Matrix 최종 업데이트
```

### 6.3 공유 형식

무거운 형식은 불필요하다. 태스크 코멘트에 아래 정도면 충분.

```md
## State Discovery
- **발견된 상태**: `error:rate-limit` (API 요청 횟수 초과)
- **조건**: 분당 100회 초과 요청 시
- **제안하는 화면 처리**: 인라인 경고 + 재시도 타이머 표시
- **API 응답**: `429 Too Many Requests` + `{ retryAfter: seconds }`
- **영향**: FE 화면 처리 추가, BE는 이미 구현됨
- **cc**: @designer @pm
```

### 6.4 State Discovery 빈도가 높은 경우

한 기능에서 State Discovery가 **3회 이상** 발생하면, 이는 Kick-off가 부실했다는 신호다.

* 해당 기능의 Kick-off 과정을 회고한다
* 빠진 패턴(Layer 2~4 누락, 에러 세분화 부족 등)을 식별한다
* 다음 Kick-off에 해당 패턴을 체크리스트에 추가한다

## 7. 다른 규정과의 연결

### 7.1 규정 ② (API 계약) 연결

State Matrix의 **"API 응답" 칸**이 API 스펙의 근거가 된다.

| State Matrix | API 스펙 (OpenAPI) |
|-------------|---------------------|
| `empty` → `{ items: [], total: 0 }` | 응답 스키마의 `items: { type: array }`, `total: { type: integer, minimum: 0 }` |
| `error:business` → `{ error: { code: "LIMIT_EXCEEDED" } }` | 에러 응답의 `code` enum에 `LIMIT_EXCEEDED` 포함 |
| `unauthorized` → `401` | 엔드포인트 응답에 `401` 정의 |

**자동 검증**: State Matrix의 에러 상태에 정의된 에러 코드가 API 스펙의 에러 응답 enum에 존재하는지 CI에서 교차 검증한다.

### 7.2 규정 ③ (UI 변경) 연결

State Matrix의 **필수 상태**가 Screen Story의 필수 케이스를 결정한다.

| State Matrix 필수 상태 | Screen Story 필수 케이스 |
|------------------------|--------------------------|
| `loading` | `Screens/FeatureName/Loading` |
| `empty` | `Screens/FeatureName/Empty` |
| `error` (1종 이상) | `Screens/FeatureName/Error` |
| `success` | `Screens/FeatureName/Default` |

**자동 검증**: PR에 Screen Story가 포함될 때, 해당 화면의 State Matrix에 정의된 필수 상태가 Screen Story에 모두 존재하는지 Claude Code hook이 확인한다.

## 8. AI 자동화

### 8.1 자동화 항목

| 항목 | 트리거 | 동작 | 도구 |
|------|--------|------|------|
| State Matrix 템플릿 생성 | 이슈/태스크 생성 시 라벨 `needs:state-matrix` | Layer 1 필수 상태 + 기능 설명 기반으로 Layer 2~4 해당 여부 제안 | Claude Code |
| 필수 상태 누락 탐지 | PR 생성 시 (Screen Story 포함 PR) | State Matrix의 필수 상태 vs Screen Story 케이스 비교 → 누락 시 경고 코멘트 | Claude Code hook |
| State Matrix ↔ API 에러 코드 불일치 | API 스펙 또는 State Matrix 변경 시 | 에러 코드 교차 검증 → 불일치 시 PR 코멘트 | CI script |
| Kick-off 체크리스트 검증 | State Matrix 문서 커밋 시 | 필수 항목 누락 여부 자동 체크 | Claude Code hook |

### 8.2 AI가 하지 않는 것

* 상태의 **화면 처리를 결정**하는 것 (디자이너의 판단)
* 비즈니스 특수 상태를 **자동으로 도출**하는 것 (도메인 지식 필요)
* Kick-off를 **대체**하는 것 (사람 간 합의가 핵심)

AI는 "빠진 게 있다"는 것을 알려줄 뿐, "무엇을 채워야 하는지"는 사람이 판단한다.

* 파일명: `{feature-name}.md` (kebab-case)

## 10. 예외 처리

### 10.1 긴급 핫픽스

운영 장애 등 긴급 상황에서는 State Matrix 없이 착수 가능.

* [ ] 착수 후 **1영업일 이내** State Matrix 소급 작성
* [ ] 소급 작성 시 실제 구현된 상태를 기준으로 작성 (있는 그대로)
* [ ] 태스크에 `hotfix:state-matrix-pending` 라벨 부여 → 소급 완료 시 제거

### 10.2 프로토타입/PoC

검증 목적의 프로토타입에서는 State Matrix 대신 **"미정의 상태 목록"**으로 대체.

```md
## Known Undefined States
- [ ] 에러 처리 미구현 (전체)
- [ ] 빈 상태 화면 미구현
- [ ] 권한 분기 미구현
```

프로토타입이 정식 기능으로 전환될 때, 반드시 State Matrix를 작성한다.

### 10.3 단계적 구현

큰 기능을 여러 PR로 나눌 때, 첫 PR에서 모든 상태를 구현하지 않아도 된다. 단, **State Matrix는 전체 상태를 포함**하고, 각 PR에서 어떤 상태를 구현하는지 명시한다.

```md
## Implementation Plan
- PR 1: `success` + `loading` (기본 흐름)
- PR 2: `empty` + `error` (엣지 케이스)
- PR 3: `unauthorized` + `forbidden` (권한 분기)
```

## 11. 점검 지표

| 지표 | 측정 방법 | 목표 |
|------|-----------|------|
| **State Discovery 빈도** | 기능당 개발 중 발견된 미정의 상태 수 | **평균 2건 이하** |
| **Kick-off 체크리스트 통과율** | 첫 시도에 통과한 비율 | **80% 이상** |
| **State Matrix 커버리지** | State Matrix가 있는 기능 / 전체 신규 기능 | **90% 이상** |
| **소급 작성 기한 준수율** | 핫픽스 후 1영업일 이내 작성 비율 | **100%** |

## 부록. 실전 예시 — "사용자 목록 화면"

> 팀이 처음 State Matrix를 작성할 때 참고할 수 있는 완성된 예시.

```md
# State Matrix: 사용자 목록 화면 (/admin/users)

> 작성자: @fe-dev
> 최종 확인일: 2025-03-08
> 관련 태스크: PROJ-1234
> 확인: 기획(✅ @pm) · 디자인(✅ @designer) · FE(✅ @fe-dev) · BE(✅ @be-dev)

## Layer 1. 데이터 상태

| 상태 | 조건 | 화면 처리 | API 응답 | 비고 |
|------|------|-----------|----------|------|
| `loading` | 최초 진입 시 | 테이블 스켈레톤 (5행) | — | |
| `loading:more` | 스크롤 하단 도달 시 | 하단 인라인 스피너 | — | 무한스크롤 |
| `empty` | 사용자 0명 | 빈 상태 일러스트 + "등록된 사용자가 없습니다" + 사용자 초대 CTA | `{ users: [], total: 0, cursor: null }` | |
| `empty:filtered` | 검색/필터 결과 0건 | "검색 결과가 없습니다" + 필터 초기화 버튼 | `{ users: [], total: 0, cursor: null }` | `empty`와 구분 |
| `success` | 사용자 1명 이상 | 사용자 테이블 (아바타, 이름, 이메일, 역할, 상태) | `{ users: [...], total: N, cursor: "..." }` | |
| `error:network` | 네트워크 단절 | 전체 에러 화면 + 재시도 버튼 | timeout | |
| `error:server` | 5xx | 전체 에러 화면 + 재시도 버튼 | `500` | |

## Layer 2. 권한·인증 상태

| 상태 | 조건 | 화면 처리 | API 응답 |
|------|------|-----------|----------|
| `unauthorized` | 미로그인 | 로그인 페이지 리다이렉트 | `401` |
| `forbidden` | 로그인됨 + admin 아님 | "관리자 권한이 필요합니다" 안내 화면 | `403` |

## Layer 3. 입력·인터랙션 상태

| 상태 | 조건 | 화면 처리 |
|------|------|-----------|
| `search:typing` | 검색어 입력 중 | 300ms 디바운스 후 자동 검색 |
| `search:loading` | 검색 요청 중 | 테이블 위 인라인 스피너 (기존 데이터 유지) |

## Layer 4. 비즈니스 특수 상태

> N/A — 이 화면에 도메인 특수 상태 없음

## 상태 전이

loading → success | empty | error
success → search:loading (검색 시) → success | empty:filtered
success → loading:more (스크롤 시) → success
error → loading (재시도 시)
empty:filtered → search:loading (검색어 변경 시) → success | empty:filtered
```
