# 프로덕트 변경 안전망 — 1장 요약

> 팀 도입 발표용. "왜 하는지, 나한테 뭐가 달라지는지"를 5분 안에 전달.

---

## 한 줄 요약

**사람이 반복적으로 놓치는 것을 시스템이 먼저 짚어주는 환경을 만든다.**

---

## 우리가 겪고 있는 문제

```
기획서 나옴 → 개발 시작 → "이 상태일 때 화면 어떻게 하죠?"
                        → "스웨거랑 실제 응답이 달라요"
                        → "토큰 바꿨는데 어디가 깨졌는지 모르겠어요"
                        → "리뷰가 3일째 안 달려요"
```

매번 사람이 기억하고 확인해야 하는 것들이다. 바쁘면 빠진다. 빠지면 나중에 더 큰 비용으로 돌아온다.

---

## 해결 방법: 세 개의 규정

```
언제?           무엇을?                    누가 신경 쓰나?
─────          ────────                   ────────────
착수 전         ① 상태 케이스 사전 정의      기획·디자인·FE·BE
개발 중         ② API 계약 관리             FE·BE
PR/머지 시      ③ UI 변경 관리              디자인·FE
```

### ① 상태 케이스 사전 정의

개발 시작하기 전에 "이 화면에서 사용자가 만날 수 있는 모든 상태"를 표 하나로 정리한다.

loading, empty, error, success — 이 네 가지는 어떤 화면이든 무조건 정의한다. "나중에 하죠"는 안 된다.

**나한테 달라지는 것**: 기능 착수 전에 State Matrix라는 표를 채우고, 기획·디자인·FE·BE 중 3명 이상이 확인해야 개발을 시작할 수 있다.

### ② API 계약 관리

스웨거가 코드에서 자동 생성되고, API가 바뀌면 자동으로 감지된다.

nullable 정확히 표시, 에러 구조 통일, 페이지네이션 명시 — 이 세 가지가 최소 기준이다.

**나한테 달라지는 것**:
- **BE**: 컨트롤러에 어노테이션을 성실히 달아야 한다. 대신 스펙 문서를 따로 쓸 필요가 없어진다.
- **FE**: 스웨거만 보고 연동한다. API 타입은 자동 생성된다. "이 필드 nullable이에요?" 같은 질문을 안 해도 된다.
- **Breaking 변경**: FE 리뷰어 승인 없이 머지할 수 없다.

### ③ UI 변경 관리

토큰/컴포넌트/화면을 바꾸면, 뭐가 영향받는지 자동으로 리포트가 나오고, VRT로 시각적 변화를 확인한다.

**나한테 달라지는 것**:
- **디자이너**: VRT diff를 보고 의도한 변화가 맞는지 승인한다. (scope=wide일 때)
- **FE**: PR에 VRT 리포트 링크와 Impact 섹션을 채운다. 대신 "이거 다른 데 영향 없어요?"라는 질문에서 해방된다.

## AI가 하는 일

AI(Claude Code)는 **확인하고 알려주는 역할**이다. 결정은 사람이 한다.

```
AI가 하는 일                          사람이 하는 일
──────────                           ──────────
"이 PR에 VRT 링크가 빠졌어요"          "이 VRT diff는 의도한 거야"
"이 API 변경은 breaking이에요"         "FE랑 합의하고 마이그레이션 가이드 쓸게"
"State Matrix에 error 상태가 없어요"   "이 화면은 에러가 이렇게 보여야 해"
"이 토큰 바꾸면 3개 컴포넌트 영향받아요" "그 3개 확인했고, 의도한 변경이야"
```

## 도입 일정 (12주)

| 주 | 단계 | 핵심 |
|----|------|------|
| 1~2 | 기반 | PR 템플릿 + CI 기본 설정 |
| 2~4 | UI 규정 | 토큰 관리 + VRT 도입 |
| 4~6 | API 규정 | 스웨거 자동 생성 + 린트 + 타입 코드젠 |
| 6~8 | 상태 케이스 규정 | State Matrix 템플릿 + Kick-off 체크리스트 |
| 8~12 | 통합 | 세 규정 연결 + 최적화 |

한 번에 다 하지 않는다. 한 단계씩, 팀이 소화할 수 있는 속도로.

---

## 이건 규칙이 아니라 안전망이다

- 예외는 막지 않는다. 단, 추적한다. (2 스프린트 이내 정상화)
- 분기마다 규정 자체를 리뷰한다. 안 쓰는 체크는 제거한다.
- "이 규칙 왜 있는지 모르겠다"가 2명 이상이면 즉시 재검토한다.

규정이 팀을 느리게 만들고 있다면, 그건 규정이 잘못된 것이다.
