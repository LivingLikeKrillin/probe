# PR 범위 규칙

## 핵심 원칙
- 하나의 PR에는 하나의 논리적 변경만 담는다
- 파일 수가 아니라 **관심사의 응집도**가 기준이다
- 정상 범위일 때는 아무 말도 하지 않는다

## 작업 중 확인
- 하나의 지시를 완료한 후, 현재까지의 변경 파일이 하나의 관심사에 속하는지 확인한다
- 새 지시를 받을 때, 기존 변경과 관심사가 달라지면 사용자에게 알린다
  - "현재 변경(User CRUD)과 다른 관심사(인증 설정)입니다. 별도 PR로 분리할까요?"
- PR 생성 직전에 `karax check`를 실행하여 최종 확인한다

## 분할 제안 시
- 어떤 파일이 어떤 그룹에 속하는지 구체적으로 나열한다
- 머지 순서를 제안한다 (의존 관계 고려)
- 강제하지 않고 제안한다 — 최종 판단은 개발자가 한다

## 경고하지 않는 경우
- 단일 도메인의 CRUD (entity + repository + service + controller + dto + test)
- 페이지 단위 기능 (page + 전용 component + hook + test)
- 리팩터링 (동일 모듈 내 구조 변경)
